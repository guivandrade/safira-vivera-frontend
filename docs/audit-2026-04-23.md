## 1. Bugs críticos (P0)

### 1.1 Rotas `(dashboard)` não são protegidas no servidor
[`src/app/(dashboard)/layout.tsx`](src/app/(dashboard)/layout.tsx) · **[middleware.ts não existe]**

Não há `middleware.ts` em nenhum nível do repo. Qualquer usuário pode digitar `/dashboard`, `/campanhas`, `/criativos`, `/configuracoes/clinica` diretamente na URL — **o Next renderiza a página**. A proteção vem só depois: quando a primeira chamada ao backend retorna 401, o interceptor do `api-client` (linha 88) redireciona pra `/login`.

**Impacto real:**
- Shape da UI (menus, layout, feature flags visíveis no HTML) vaza antes do redirect.
- Usuário com conexão lenta vê o dashboard "piscar" antes de ser mandado pro login. Parece bug.
- Se o backend algum dia responder 200 num endpoint público por engano, toda a UI sobe sem auth.
- Defesa em profundidade ausente: hoje o frontend depende 100% do backend responder 401.

**Correção:** `middleware.ts` na raiz verificando presença do `access_token` cookie (requer mover tokens pro cookie httpOnly — ver §2.1) **ou** `access_token` no header de request (se mantido em localStorage, só o client consegue ler, então a verificação server-side fica cega e isso vira argumento a mais pra migrar pra cookie).

### 1.2 Página de login ignora dark mode e o design system
[`src/app/login/page.tsx:65-180`](src/app/login/page.tsx#L65-L180)

Login usa cores hardcoded que **não existem no design system da Safira** e **nenhum prefixo `dark:`**:
- `from-blue-50 to-white`, `bg-white`, `text-gray-900`, `text-gray-700`, `text-gray-500`, `text-gray-600`
- `bg-indigo-600`, `focus:ring-indigo-500`, `hover:text-indigo-600`

Enquanto isso, o resto do app usa tokens semânticos: `bg-surface`, `text-ink`, `border-line`, `bg-accent` (que é o Sapphire Gold `#F4C03E` ou Sapphire Blue `#0A3DA4` conforme definido em [`safira-standards.md`](../../../.claude/ref/safira-standards.md)).

**Efeito:**
- Usuário com dark mode ativo: a tela de login inteira continua branca. Parece outro produto.
- Branding inconsistente — login é a primeira impressão e é onde a identidade Safira some.
- O theme init script em [`src/app/layout.tsx:21`](src/app/layout.tsx#L21) aplica `dark` no `<html>` no primeiro paint. Na login, isso não altera nada porque os estilos estão hardcoded.

**Severidade P0** porque é a primeira tela que qualquer usuário vê, e é exatamente a que está fora do padrão visual.

### 1.3 `window.location.href = response.data.authUrl` sem validar shape
[`src/components/features/campaigns/ConnectionStatusCards.tsx:42`](src/components/features/campaigns/ConnectionStatusCards.tsx#L42) · [`CampaignsDashboard.tsx:99`](src/components/features/campaigns/CampaignsDashboard.tsx#L99)

```ts
const response = await apiClient.get<{ authUrl: string }>('/integrations/google-ads/oauth/authorize');
window.location.href = response.data.authUrl;
```

Se o backend retornar `{}` ou `{ authUrl: null }` em caso de erro 200 (bug comum em APIs mal desenhadas), a expressão navega pra `"undefined"` literal, quebrando o fluxo de onboarding do Google Ads sem mensagem clara. Não é exploit de segurança (é backend interno), mas é bug de robustez. Guard simples:
```ts
if (!response.data?.authUrl) throw new Error('authUrl ausente na resposta');
```

## 2. Segurança

### 2.1 Tokens em `localStorage` — vulnerável a XSS
[`src/lib/api-client.ts:15,29,39,41,72,73`](src/lib/api-client.ts) · [`login/page.tsx:49-51`](src/app/login/page.tsx#L49)

`access_token`, `refresh_token` e `user` vivem em `localStorage`. Qualquer script rodando na origem consegue ler:
```js
localStorage.getItem('access_token')
```

**Por que isso importa pra Vívera especificamente:**
- O backend serve dados de campanhas da clínica + dados de pacientes (conversões).
- Token roubado = atacante autentica como Vera e extrai tudo.
- **Risco LGPD direto.**

**Vetor atual realista:**
- Usamos `<img src={row.thumbnailUrl}>` com URL vinda do Meta ([`CreativesPage.tsx:413`](src/components/features/creatives/CreativesPage.tsx#L413)). Se um dia o Meta CDN for comprometido ou um criativo contiver SVG com JS inline, o site fica exposto.
- Sem CSP (ver §2.2), qualquer `<script>` injetado roda livre.

**Mitigação:**
- Backend deve aceitar auth via cookie `httpOnly; Secure; SameSite=Strict` + CSRF token.
- Frontend para de tocar no token — só o browser envia automaticamente.
- Migration path requer mudança coordenada backend + frontend.

Se não puder migrar pra cookie no curto prazo, pelo menos: CSP rigorosa (§2.2), subresource integrity em scripts externos, e nenhum `dangerouslySetInnerHTML` com conteúdo user-provided (hoje só existe 1 e é interno — ok).

### 2.2 Sem Content Security Policy
[`next.config.js:24-44`](next.config.js#L24)

`next.config.js` seta 3 headers: `X-Content-Type-Options: nosniff`, `X-Frame-Options: SAMEORIGIN`, `X-XSS-Protection: 1; mode=block`. **Falta:**
- `Content-Security-Policy` (o único que de verdade previne XSS moderno — `X-XSS-Protection` é legacy e ignorado por navegadores atuais)
- `Strict-Transport-Security`
- `Referrer-Policy`
- `Permissions-Policy`

Mínimo sugerido pra este app (Meta/Google APIs + localhost dev):
```
Content-Security-Policy: default-src 'self'; script-src 'self' 'unsafe-inline'; img-src 'self' https: data:; connect-src 'self' https:; font-src 'self' data:; frame-ancestors 'none'
Strict-Transport-Security: max-age=31536000; includeSubDomains
Referrer-Policy: strict-origin-when-cross-origin
Permissions-Policy: camera=(), microphone=(), geolocation=()
```

`'unsafe-inline'` em `script-src` ainda é necessário por causa do theme init script. Pode ser endurecido depois com nonce.

### 2.3 Open redirect potencial via `returnUrl`
[`src/app/login/page.tsx:30,53`](src/app/login/page.tsx#L30)

```tsx
const returnUrl = searchParams.get('returnUrl') || '/campanhas';
// ...
router.push(returnUrl);
```

`router.push` do App Router é restrito a rotas internas — URL absoluta externa é ignorada no client-side. Mas valores maliciosos como `//evil.com/path` ou `/\\evil.com` podem ser interpretados como paths relativos em alguns browsers/versões antes do Next sanitizar.

**Impacto:** baixo (protecção implícita do Next), mas é um padrão que vira vetor real quando alguém adiciona `window.location.href = returnUrl` depois. Guard barato:
```ts
const safeReturnUrl = returnUrl.startsWith('/') && !returnUrl.startsWith('//') ? returnUrl : '/campanhas';
```

### 2.4 `remotePatterns: hostname: '**'`
[`next.config.js:13-18`](next.config.js#L13)

```js
images: {
  remotePatterns: [{ protocol: 'https', hostname: '**' }],
},
```

Aceita imagens de **qualquer host HTTPS** via `next/image`. Na prática o código não usa `next/image` nas thumbnails (usa `<img>` puro em [`CreativeThumb`](src/components/features/creatives/CreativesPage.tsx#L406)), então o risco atual é apenas: se alguém adicionar `<Image src="https://evil.com/xss.svg">` no futuro, passa. Melhor restringir agora:
```js
remotePatterns: [
  { protocol: 'https', hostname: '*.fbcdn.net' },
  { protocol: 'https', hostname: 'scontent.cdninstagram.com' },
  { protocol: 'https', hostname: 'lookaside.fbsbx.com' },
  // + o que o Google Ads puxar
]
```

### 2.5 15 vulnerabilidades no `npm audit`
`npm audit` reporta **5 moderate, 10 high**, todas em transitives via `eslint-config-next@14` (`glob` → `minimatch`, `@typescript-eslint/*@6`). Não há vulnerabilidade em deps de runtime — todas são em devDeps. Fix exige upgrade major (`eslint-config-next@16`), que é compatível mas requer revisão dos ajustes de lint.

Não é P0, mas reportar verde em `npm audit` é a linha de base.

## 3. UX / UI

### 3.1 Alt text vazio em criativos (image é conteúdo, não decorativo)
[`src/components/features/creatives/CreativesPage.tsx:413`](src/components/features/creatives/CreativesPage.tsx#L413)

```tsx
<img src={row.thumbnailUrl} alt="" ... />
```

O comentário `// eslint-disable-next-line @next/next/no-img-element` indica uso consciente de `<img>` em vez de `<Image>`. Mas `alt=""` marca a imagem como **decorativa** — leitores de tela pulam. Em /criativos, a imagem **é o conteúdo principal** (é a própria peça do anúncio). `alt={row.name}` ou `alt={\`Criativo ${row.name}\`}` resolve.

Também: [`CreativeThumb`](src/components/features/creatives/CreativesPage.tsx#L406) não usa `next/image` nem `loading="lazy"` — numa galeria com 50-100 criativos, todas as imagens baixam de uma vez.

### 3.2 Nenhuma página `not-found.tsx` nem `error.tsx`
Não existe [`src/app/not-found.tsx`](src/app/not-found.tsx) nem [`src/app/error.tsx`](src/app/error.tsx) nem variantes em route groups.

- URL errada → Vera vê a tela default do Next ("404 | This page could not be found"). Sem branding Safira, sem navegação pra voltar pra `/dashboard`.
- Erro em Server Component → tela default de erro do Next. O `ErrorBoundary` do PR #35 cobre só Client Components dentro do `(dashboard)/layout.tsx`.

Ambas são 20 linhas de código cada. `error.tsx` é **obrigatório** por boa prática em App Router.

### 3.3 Mensagens de erro sem distinguir tipo de falha
[`src/app/login/page.tsx:54-58`](src/app/login/page.tsx#L54)

```tsx
const message =
  err?.response?.data?.message ||
  (err?.response?.status === 401 ? 'Email ou senha incorretos' : 'Erro ao fazer login');
```

Falha de rede (sem conexão, backend down, CORS) cai em "Erro ao fazer login" — indistinguível de 500. Vera fica tentando a senha de novo pensando que digitou errado.

Sugestão:
- `!err?.response` (erro de rede) → "Não conseguimos contato com o servidor. Verifique sua conexão."
- `status >= 500` → "O sistema está temporariamente indisponível. Tente em alguns minutos."
- `status === 401` → "Email ou senha incorretos."

### 3.4 Botão "Mostrar senha" sem feedback de aria-pressed
[`src/app/login/page.tsx:111-117`](src/app/login/page.tsx#L111)

```tsx
<button type="button" onClick={...} aria-label={showPassword ? 'Ocultar senha' : 'Mostrar senha'}>
```

`aria-label` muda, mas para a11y é mais idiomático usar `aria-pressed={showPassword}` num toggle. Minor.

### 3.5 `setInterval` em `FreshnessIndicator` sem pausa
[`src/components/ui/FreshnessIndicator.tsx:34`](src/components/ui/FreshnessIndicator.tsx#L34)

```ts
const i = setInterval(() => force((n) => n + 1), 30_000);
```

Força re-render a cada 30s pra atualizar "há X minutos". Em toda a árvore que consome, isso dispara reconciliação. Não é hot path (só atualiza timestamp), mas:
- Se a aba está oculta, o timer continua rodando.
- Se a página tem 3 `FreshnessIndicator`, são 3 timers independentes.

Mitigação barata: pausar quando `document.visibilityState !== 'visible'`. Retorno: economia de CPU/bateria em mobile.

### 3.6 `CampaignsDashboard` sem refresh automático
Audit anterior (§3 médio impacto) já mencionou. Continua aberto — dashboards de mídia paga geralmente têm auto-refresh opcional (3-5min). Feature pequena; vale pro Sprint 2.

### 3.7 Loading states em ações rápidas (`<100ms`)
`<Button>` de submit de login, disconnect Google, etc. mostram texto "Entrando...", "Desconectando..." imediatamente. Quando a resposta volta em <100ms, o texto pisca — UX pior que nenhum indicador. Padrão melhor: delay de 200-300ms antes de mostrar spinner/texto de loading (pattern conhecido: "lazy loading indicator").

## 6. Data / State

### 6.1 Invalidação de cache pós-mutação ausente
[`src/hooks/use-creatives.ts`](src/hooks/use-creatives.ts) · [`use-local-geography.ts`](src/hooks/use-local-geography.ts) · [`use-keywords.ts`](src/hooks/use-keywords.ts)

Hooks com cursor pagination (PR #34). Se amanhã surgir mutação que edita/deleta um desses (ex: pausar criativo, renomear campanha), **nenhuma `invalidateQueries` está definida**. Usuário vai continuar vendo o estado antigo até refresh manual.

Hoje não há mutações dessas — então não é bug atual. Mas é um **buraco arquitetural**: quando chegar a primeira write-op (audit anterior fala de "6 meses — plataforma com write-ops"), é o primeiro ponto que quebra.

### 6.2 `staleTime` inconsistente entre hooks
Sem comparar todos os hooks, já dá pra ver:
- Alguns usam `staleTime: 30_000` (30s)
- Outros `staleTime: 5 * 60 * 1000` (5min)
- `usePreferences` usa `5 * 60 * 1000`

Sem documento que diga "qual stale time pra qual tipo de dado". Convenção sugerida:
- Real-time (insights principais): 60s
- Settings de usuário: 5min
- Integrações (status de conexão): 30s
- Referência (categorias, enums): 1h

### 6.3 Optimistic update em `useUpdatePreferences` pode divergir
[`src/hooks/use-preferences.ts:41-52`](src/hooks/use-preferences.ts#L41)

```ts
onMutate: async (next) => {
  await qc.cancelQueries({ queryKey: ['preferences'] });
  const previous = qc.getQueryData<UserPreferences>(['preferences']);
  qc.setQueryData(['preferences'], next);
  return { previous };
},
onError: (_err, _vars, ctx) => {
  if (ctx?.previous) qc.setQueryData(['preferences'], ctx.previous);
},
```

Está correto. **Mas:** se o usuário faz 3 mudanças rápidas (reorder coluna → toggle hide → reorder outra) e só a 2ª falha, o `ctx.previous` é o estado anterior à 2ª — sobrescreve as mudanças 1 e 3. Baixíssima probabilidade de acontecer na UI real (operações são pontuais), mas o debounce de 500ms que o hook usa atenua. Ok por enquanto.

## 9. Visão sobre o sistema hoje

### O que está funcionando bem
- **Design system maduro e consistente** (exceto login).
- **Estrutura de pastas limpa** — route groups, separação features/ui/layout, hooks por endpoint, stores por domínio.
- **Auth flow robusto** depois do PR #33 — refresh rotation + single-flight resolve o problema clássico de corrida em 401.
- **Data layer coerente** — filtro global aplicado uniformemente, KPIs e insights calculam do mesmo `campaigns[]`.
- **UX diferencial** — Cmd+K, drag-and-drop, URL sync de filtros, saved views, freshness indicator.
- **Confiabilidade saltou** com PR #35 — ErrorBoundary + 87 testes é base sólida.

### Como o sistema funciona hoje (fluxo real)
1. Vera abre URL → Next serve HTML do `(dashboard)/layout`.
2. Client component hidrata, lê `access_token` do localStorage, começa requisições.
3. Stores Zustand reidratam do localStorage (filtros, layout, preferências de coluna).
4. React Query dispara 4-6 queries em paralelo: insights, clinic, integrations, preferences.
5. UI aparece em estados progressivos — skeleton → dados parciais → dados completos.
6. Vera interage via sidebar/tabs/filter bar → URL sync atualiza, stores persistem, preferences sincronizam com backend debounced.

**Isso está bom.** Os pontos cegos estão nas **bordas**: primeira tela (login), resposta a erros (sem UX de rede caindo, sem 404 com branding), proteção de rotas, e a hipótese de que o backend sempre responde bonitinho.

### Como deveria funcionar (visão curta)
- **Auth:** tokens em cookie httpOnly. Middleware redireciona no servidor antes de hidratar a UI. Login e dashboard respeitam o mesmo theme.
- **Resiliência:** CSP ativa. Sentry plugado. `error.tsx` e `not-found.tsx` com branding. ErrorBoundary reporta erros silenciosos.
- **Observabilidade:** métrica de "tempo até primeiro dado visível" medida em produção, não só em dev.
- **Performance:** Recharts lazy. Imagens de criativos com `next/image` + lazy load. Bundle inicial < 200kB.
- **Qualidade:** zero `catch (err: any)`. Migrations de store testadas. Componentes > 300 LOC quebrados em partes.
- **Cobertura de teste:** hooks críticos (useCampaignInsights, useColumnOrder) testados. 1 E2E de login + dashboard.

Nada disso é inventar produto novo — é **completar o que já está 80% feito**.

### Como deveria funcionar (visão longa, 3-6 meses)
Já bem capturado no audit anterior (§6 e §7). Mantenho a priorização: alertas proativos e goals antes de expandir features; write-ops como próxima fronteira; multi-tenant real quando fechar outro cliente.
